## The Question Nobody Asks

You open Google Authenticator. A 6 digit code appears. 30 seconds later it expires. A new one appears.

Your phone generated that code. The server validates it. They never communicated to agree on that code. No API call. No database lookup. No stored OTP.

How do both sides arrive at the exact same number independently?

Most engineers use this every day and have never stopped to think about it. Here's what's actually happening.

---

## Not All OTPs Are the Same

Before going deeper, it's worth knowing that there are two completely different types of OTPs that look identical to the user.

**SMS OTP — the code your bank sends you**

- Server generates a random number
- Stores it in a database with an expiry timestamp
- Sends it to you via SMS
- Validates by looking it up in the database
- Deletes it after use or expiry

This is simple. Just a random number stored temporarily.

**TOTP — Time Based One Time Password (Google Authenticator)**

- Nothing is stored after initial setup
- Generated by math on two shared inputs
- Works completely offline
- Expires every 30 seconds automatically

This is the interesting one. And this is what the rest of this post is about.

Two completely different mechanisms. Identical experience to the user.

---

## How TOTP Works

For your phone and the server to independently generate the same number, they need two things agreed upon in advance.

### Thing 1: A Shared Secret

When you first set up Google Authenticator — you scan a QR code. That QR code contains a secret key. Now both your phone and the server have the same secret key stored.

That's the one time they communicate. After that — never again.

### Thing 2: Time

Both your phone and the server know what time it is. And time is the same for everyone.

So now both sides have:

- Same secret key
- Same current time

They run the same mathematical formula on those two inputs and independently arrive at the same 6 digit number.

```
Phone:  HMAC(secret_key + current_time_window) → 6 digit code
Server: HMAC(secret_key + current_time_window) → same 6 digit code
```

No communication needed. No storage needed. Just math on two shared inputs.

---

## What About Clock Drift?

Phone clocks drift slightly over time. Not by minutes — but by a few seconds. Google Authenticator works offline, so it uses the phone's local clock.

If your phone is 10 seconds behind the server, the code you generate might be from a slightly different time window than what the server expects.

The solution: the server doesn't just validate the current 30 second window. It also accepts codes from the previous window and the next window.

```
Accepted windows:
- Code from 30 seconds ago
- Current code
- Code for next 30 seconds
```

That's a 90 second validity buffer in practice — even though the user sees 30 seconds. Enough buffer for clock drift without meaningfully compromising security.

---

## Can You Reuse a Code?

You generate a code. You use it to log in. The 30 second window hasn't expired. Can you use the same code again?

No. And here's how that's prevented without storing the OTP.

The server keeps one tiny record per account — not the code itself, but whether a code from this time window has already been used.

Just a boolean: _"Has this account already authenticated in the current 30 second window?"_

If yes — reject. If no — allow and mark as used.

That's the only thing stored. Not the OTP. Not the secret. Just "this window is consumed."

---

## The Full Picture

```
Setup — one time only:
Server and phone share a secret key via QR code scan

Every 30 seconds:
Phone: secret + current time → math → 6 digit code
Server: secret + current time → same math → same 6 digit code

Validation:
User enters code
Server checks current window + previous + next (clock drift buffer)
Server checks if this window already used for this account
If match and not reused → authenticated
```

## Where This Can Break

### The Secret Key is Everything

The entire security of TOTP rests on one thing — keeping the secret key secret.

The mathematical formula is public knowledge. TOTP is an open standard — anyone can read the spec. If an attacker has your secret key, they can generate valid codes forever without needing your phone.

This is why:

- QR codes should never be screenshotted and stored in cloud photo libraries
- Secret keys should never be transmitted over unencrypted channels
- If you lose your phone, immediately regenerate your TOTP secret on every service

### TOTP Doesn't Protect Against Phishing

An attacker builds a fake Gmail login page. You enter your email, password, and TOTP code. The attacker instantly relays all three to real Gmail in real time. They're in.

TOTP didn't help because you willingly gave the code to the attacker. The code was valid. The math was correct. The security failed at the human layer, not the technical one.

This is why hardware security keys like YubiKey exist — they verify the domain they're talking to. A fake Gmail page can't trick a YubiKey because the domain doesn't match.

## What This Teaches You as an Engineer

**Cryptography solves storage problems**

TOTP is elegant because it replaces a storage and transmission problem with a math problem. Instead of "how do we securely store and transmit OTPs" — it asks "how do we make both sides independently compute the same value." That shift in thinking is worth internalizing.

**The only secret is the secret**

In well designed cryptographic systems, the algorithm is always public. Only the key is secret. If your system's security depends on keeping the algorithm hidden — that's a red flag. Security through obscurity is not security.

**Every system has a human layer**

TOTP is mathematically sound. But phishing bypasses it entirely because humans are in the loop. When designing security systems, technical strength is necessary but not sufficient. The human layer needs to be part of the threat model.

---

_The most elegant security systems don't store secrets. They make secrets computable from things both sides already know._
